# Отладка ошибки 400 Bad Request

## Временное решение: Включить DEBUG для просмотра детальной ошибки

Чтобы увидеть точную причину ошибки 400, временно включим DEBUG:

### 1. В Railway Variables:

1. Откройте сервис up11 → Variables
2. Найдите переменную `DEBUG`
3. Измените значение на: `True`
4. Сохраните
5. Перезапустите сервис (три точки → Restart)

### 2. После перезапуска:

1. Обновите страницу в браузере
2. Теперь вы увидите детальную ошибку Django вместо просто "Bad Request (400)"
3. Ошибка покажет точную причину проблемы

### 3. Возможные причины ошибки 400:

**A. Проблема с ALLOWED_HOSTS:**
- Ошибка: "DisallowedHost at /"
- Решение: Проверьте, что ALLOWED_HOSTS включает ваш домен

**B. Проблема с CSRF:**
- Ошибка: "CSRF verification failed"
- Решение: Добавьте CSRF_TRUSTED_ORIGINS

**C. Проблема с базой данных:**
- Ошибка: "relation does not exist" или "table does not exist"
- Решение: Миграции не применены полностью

**D. Проблема с настройками:**
- Ошибка: "ImproperlyConfigured"
- Решение: Проверьте все переменные окружения

### 4. После выяснения причины:

1. Исправьте проблему
2. Верните `DEBUG=False` в Variables
3. Перезапустите сервис

## Альтернатива: Проверка логов Railway

1. Откройте сервис up11 → Deployments
2. Нажмите "View logs"
3. Прокрутите до конца логов
4. Ищите ошибки Django (обычно красным цветом)
5. Пришлите последние строки логов
